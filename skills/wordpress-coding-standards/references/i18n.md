# Internationalization (i18n)

Every string a user sees should be wrapped in a translation function with the project's text domain. This is what lets a Polish user see Polish, a Japanese user see Japanese, and the WordPress.org translation team contribute translations without you doing anything else.

Skipping i18n is one of the most common reasons plugins get rejected from the WordPress.org Plugin Directory.

## The text domain

The text domain is a unique slug that identifies the strings as belonging to your plugin or theme. It must match the slug declared in the plugin/theme header:

```php
/*
 * Plugin Name: My Awesome Plugin
 * Text Domain: my-awesome-plugin
 * Domain Path: /languages
 */
```

The text domain string in every translation call must be a **plain literal**. Never a variable, constant, or concatenation:

```php
// Good
__( 'Hello', 'my-awesome-plugin' )

// All bad — WordPress.org's translation scanner can't parse these statically
__( 'Hello', $text_domain )
__( 'Hello', MY_PLUGIN_TEXT_DOMAIN )
__( 'Hello', 'my-' . 'awesome-' . 'plugin' )
```

A common convention is to set the text domain as a project-wide search-and-replace value. When refactoring, change every literal at once — don't replace with a constant.

## Translation functions

The function name encodes three pieces of information:

1. **Return vs echo**: `__()` returns, `_e()` echoes
2. **Escape**: `esc_html_*`, `esc_attr_*`, or none
3. **Variant**: plural (`_n_`), context (`_x_`), or neither

| Function                         | Returns / Echoes | Escapes | Notes                                                                   |
| -------------------------------- | ---------------- | ------- | ----------------------------------------------------------------------- |
| `__( $text, $domain )`           | returns          | no      | The base. Use when you'll wrap the result before output                  |
| `_e( $text, $domain )`           | echoes           | no      | Avoid — almost always you should escape on output                       |
| `esc_html__( $text, $domain )`   | returns          | HTML    | Translate + escape for HTML body                                        |
| `esc_html_e( $text, $domain )`   | echoes           | HTML    | The most common form for inline output in HTML body                     |
| `esc_attr__( $text, $domain )`   | returns          | attr    | For HTML attributes                                                     |
| `esc_attr_e( $text, $domain )`   | echoes           | attr    | Echoes into an attribute                                                |
| `_n( $singular, $plural, $count, $domain )`         | returns | no | Plural form                                              |
| `_nx( $singular, $plural, $count, $context, $domain )` | returns | no | Plural with context                                   |
| `_x( $text, $context, $domain )`  | returns         | no      | With context (for disambiguating identical strings)                     |
| `_ex( $text, $context, $domain )` | echoes          | no      | Echoing variant of `_x`                                                 |
| `esc_html_x( $text, $context, $domain )` | returns | HTML | Context + HTML escape                                                  |
| `esc_attr_x( $text, $context, $domain )` | returns | attr | Context + attr escape                                                  |

**Default to the `esc_*__` / `esc_*_e` forms.** They make the escape contract explicit at the call site, and a translator can't break your escaping by introducing problematic characters.

## Common patterns

### Simple HTML body text

```php
echo '<h2>' . esc_html__( 'Plugin Settings', 'my-awesome-plugin' ) . '</h2>';

// Or, when the whole line is the translation:
echo '<h2>';
esc_html_e( 'Plugin Settings', 'my-awesome-plugin' );
echo '</h2>';
```

### Attribute values

```php
?>
<input
    type="text"
    placeholder="<?php esc_attr_e( 'Enter your API key', 'my-awesome-plugin' ); ?>"
    aria-label="<?php esc_attr_e( 'API key', 'my-awesome-plugin' ); ?>"
>
<?php
```

### Strings with placeholders

Use `sprintf` (or `printf` for echo). The format string itself is translated; placeholders are replaced with the (separately escaped) values.

A translator comment **must** appear immediately before the call, in a specific format:

```php
printf(
    /* translators: %s: user display name */
    esc_html__( 'Welcome back, %s!', 'my-awesome-plugin' ),
    esc_html( $user->display_name )
);
```

The `/* translators: ... */` comment is parsed by WordPress.org's i18n tooling and shown to translators. Without it, translators see the format string in isolation and have no idea what `%s` represents.

For multiple placeholders, number them so translators can reorder for languages with different word order:

```php
printf(
    /* translators: 1: post title 2: user display name */
    esc_html__( '%1$s by %2$s', 'my-awesome-plugin' ),
    esc_html( $post->post_title ),
    esc_html( $user->display_name )
);
```

A French translator might reorder this as `'Par %2$s : %1$s'` without you doing anything.

### Plurals

```php
$count = count( $items );

printf(
    /* translators: %d: number of items deleted */
    esc_html(
        _n(
            '%d item deleted.',
            '%d items deleted.',
            $count,
            'my-awesome-plugin'
        )
    ),
    (int) $count
);
```

`_n()` doesn't have an escape variant, so wrap the result in `esc_html()` explicitly.

Some languages have more than two plural forms (Russian, Polish, Arabic). `_n()` handles this automatically based on the language's rules — you provide singular and plural English, and the translation tooling does the rest.

### Context disambiguation

When the same English word means different things in different places, use `_x` to give translators context:

```php
// "Post" the verb
$button = esc_html_x( 'Post', 'verb: submit a comment', 'my-awesome-plugin' );

// "Post" the noun
$label  = esc_html_x( 'Post', 'noun: a blog post', 'my-awesome-plugin' );
```

Both produce "Post" in English, but a translator might choose different words for each (e.g., German *Posten* vs *Beitrag*).

### Strings with HTML

When the translatable string itself contains HTML (e.g., a `<strong>` tag or a link), don't escape the result — but think carefully about what the translator can break:

```php
printf(
    wp_kses(
        /* translators: %s: documentation link */
        __( 'See the <a href="%s">documentation</a> for details.', 'my-awesome-plugin' ),
        array(
            'a' => array(
                'href' => array(),
            ),
        )
    ),
    esc_url( 'https://example.com/docs' )
);
```

`wp_kses()` allows the `<a>` tag but strips anything else a translator might inject — defence in depth.

Better still: keep HTML out of translated strings when possible. Translators sometimes break tags, and managing it is friction for everyone:

```php
// Often simpler:
$link = '<a href="' . esc_url( 'https://example.com/docs' ) . '">'
      . esc_html__( 'documentation', 'my-awesome-plugin' )
      . '</a>';
printf(
    /* translators: %s: documentation link */
    esc_html__( 'See the %s for details.', 'my-awesome-plugin' ),
    $link // already-safe pre-built HTML — usually flagged by PHPCS; suppress with reason
);
```

The `phpcs:ignore` for the prebuilt-HTML pattern is:

```php
// phpcs:ignore WordPress.Security.EscapeOutput.OutputNotEscaped -- $link is built from already-escaped fragments.
```

## What to translate

Translate everything a user might see:

- Labels, buttons, menu items, page titles
- Admin notices, error messages, success messages
- Placeholder text in inputs
- Tooltip text, help text
- Validation messages
- Console/log messages users will see (e.g., site health debug info)
- Email subjects and bodies if your plugin sends emails

**Don't** translate:

- Internal keys, slugs, option names, capability names, hook tags, CSS classes — those are code, not content
- Strings only used in dev-only debug output that no end user will see
- Strings that are themselves technical syntax (e.g., a regex, JSON content)

When unsure: if a non-developer would see it, translate it.

## Loading the text domain

For plugins:

```php
function my_plugin_load_textdomain() {
    load_plugin_textdomain(
        'my-awesome-plugin',
        false,
        dirname( plugin_basename( __FILE__ ) ) . '/languages'
    );
}
add_action( 'init', 'my_plugin_load_textdomain' );
```

Hook to `init`, not earlier. Translation functions don't work before `init` because the textdomain isn't loaded yet — if you call `__()` during `plugins_loaded`, you get the English string regardless of the user's locale.

For plugins hosted on WordPress.org, translation files are served from WordPress.org's translation infrastructure and the `load_plugin_textdomain` call is technically optional — WordPress will load them automatically from `wp-content/languages/plugins/`. But include the call anyway for off-WordPress.org distribution and for early-version compatibility.

For themes:

```php
function my_theme_load_textdomain() {
    load_theme_textdomain(
        'my-theme',
        get_template_directory() . '/languages'
    );
}
add_action( 'after_setup_theme', 'my_theme_load_textdomain' );
```

### The `init` rule

Don't call any translation function before `init`. If your plugin registers things during `plugins_loaded` or even at file include time (custom post types, taxonomies registered with translated labels), wrap the labels in functions that get called at the right time, or use closures:

```php
// Good — labels are computed at registration time, which fires on 'init'
add_action( 'init', function () {
    register_post_type( 'my_book', array(
        'labels' => array(
            'name'          => __( 'Books', 'my-awesome-plugin' ),
            'singular_name' => __( 'Book', 'my-awesome-plugin' ),
        ),
        // ...
    ) );
} );
```

## The `Domain Path` header

In the plugin file header, declare where translations live:

```php
/*
 * Plugin Name: My Awesome Plugin
 * Text Domain: my-awesome-plugin
 * Domain Path: /languages
 */
```

The path is relative to the plugin root. WordPress uses this to find `.mo` files bundled in the plugin.

## PHPCS i18n sniffs

The `WordPress.WP.I18n` sniff catches:

- Translation calls with non-literal text domains
- Translation calls without a text domain (silently default to `default`, which is core)
- Translation calls with the wrong text domain for the project
- Strings with placeholders missing the `/* translators: */` comment
- Use of `__()` where `esc_html__()` was needed
- Double-encoding from chained `esc_html( esc_html__( ... ) )` calls

Configure the text domain in `phpcs.xml.dist` so the sniff knows what to look for:

```xml
<rule ref="WordPress.WP.I18n">
    <properties>
        <property name="text_domain" type="array">
            <element value="my-awesome-plugin"/>
        </property>
    </properties>
</rule>
```

## Checklist

- [ ] Every user-facing string is wrapped in a translation function.
- [ ] The text domain in every call is a plain string literal matching the plugin's slug.
- [ ] The text domain matches the `Text Domain` header in the plugin/theme file.
- [ ] Translation functions are called on `init` or later — never earlier.
- [ ] `load_plugin_textdomain()` (or `load_theme_textdomain()`) is hooked appropriately.
- [ ] Every `printf`/`sprintf` with translated format has a `/* translators: */` comment above it.
- [ ] Placeholders in plural strings use numbered form (`%1$s`, `%2$d`) where there's more than one.
- [ ] Translated HTML is wrapped in `wp_kses()` with an explicit allowlist, or built from already-safe fragments.
- [ ] The escape variant matches the output context (`esc_html__` for HTML body, `esc_attr__` for attributes).
